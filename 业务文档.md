# API 集成平台（动态版）— 业务说明文档

> 本文档面向**项目分享、协作与新人 onboarding**：说明平台是什么、解决什么问题、带来什么价值。技术接入细节见仓库内 `skills/` 目录下的 Skill 文档。

---

## 1. 文档目的

- 让接收本仓库的同事**快速建立共同认知**：这不是零散脚本集合，而是一套**可复用的云端 API 演示与验证环境**。
- 便于在团队内推广时，用统一话术说明**业务价值**与**使用边界**。

---

## 2. 平台定位（一句话）

**API 集成平台（动态版）** 将第三方云端能力（以 OCR、文档理解、教育场景等为主）封装为**浏览器内可切换的 Demo**：上传或传参即可调用，左侧看原始 JSON，右侧看可视化结果，支持复制与下载，便于评测、对比与演示。

---

## 3. 解决什么问题（痛点）

| 痛点 | 说明 |
|------|------|
| **验证成本高** | 产品经理或业务在验证模型/接口效果时，常需频繁编写、修改脚本，环境不统一。 |
| **工具链繁琐** | 鉴权、同步/异步、轮询、图片编码等重复劳动多，非研发同学难以独立完成。 |
| **复用性差** | 换厂商、换接口时难以沉淀标准流程，知识散落在个人电脑里。 |
| **分享与演示弱** | 原始接口返回多为 JSON，不利于汇报、竞品对比和跨角色沟通。 |

本平台通过**统一 Web 入口 + 标准化接入约定**，把「能跑通 + 能看懂 + 能展示」固化在同一套界面与流程里。

---

## 4. 平台能做什么（能力范围）

1. **多能力菜单切换**  
   左侧菜单对应多个独立 Demo 页面，每个页面面向一类云端接口或业务场景。

2. **端到端调用**  
   后端统一处理鉴权与厂商协议（含异步任务与轮询等）；前端负责参数表单与结果展示。

3. **统一结果呈现**  
   约定**左侧展示完整 JSON**，**右侧做可视化**（OCR 等场景强调在原图上叠加框线与标注）；并提供 JSON **一键复制**与**下载**，便于存档与二次分析。

4. **可维护的生命周期**  
   新增或下架某个 Demo 时，有明确的代码位置与检查步骤（见仓库内「删除 API」相关 Skill），避免菜单、路由、文件不一致。

**说明**：具体已接入的厂商与接口名称以运行后左侧菜单为准；本仓库会随业务迭代增减 Demo。

---

## 5. 核心价值与收益

| 对象 | 收益 |
|------|------|
| **产品 / 业务** | 缩短从「想试一个接口」到「看到可视化结果」的路径，支撑**模型效果验证、竞品调研、对内对外演示**。 |
| **技术 / 维护** | 前后端结构与 Skill 文档降低**接入与下架**的心智负担，修改点集中、可检索、可复用。 |
| **组织** | Demo 形态统一，便于在组内**推广为标准的验证与展示入口**，减少重复造轮子与重复沟通。 |

---

## 6. 五个 Skill 如何作用于平台

平台不是「只有一套页面代码」，而是 **「可运行的产品 + 一套可复用的操作规范（Skill）」**。五个 Skill 分别对应**地图、总流程、厂商细节、展示标准、下架闭环**；人机协作（同事或 AI 助手按 Skill 改代码）时，能保证**同一套接入方式、少踩坑、可交接**。

### 6.1 分工一览：每个 Skill 解决什么问题

| Skill | 在平台里的角色 | 作用对象（落到哪里） |
|------|----------------|----------------------|
| **前端后端架构**（`frontend-backend-architecture-skill`） | **「地图」与约定** | 说清根目录、`server.py` 路由、`index.html` 侧栏与 iframe、`demos/` 单页结构、静态资源怎么访问；**任何**接入或删除都先对齐这张「地图」，避免改错文件或漏改入口。 |
| **集成**（`integration-skill`） | **「接一个新 Demo」的总流程** | 把厂商文档变成平台内可点可用的页面：环境检查 → 后端路由与转发 → 新建 `demos/*.html` 并在 `index.html` 加菜单 → 按可视化规范做结果区；强调**参数与官方文档一致、覆盖必选与可选参数**。 |
| **厂商 API 调用**（`vendor-api-skill`） | **「怎么跟云端说话」** | 各厂商的鉴权（如百度 Token）、同步/异步、轮询间隔与「处理中」状态等；**集成 Skill 在写后端时**要查阅本 Skill，保证调用方式符合厂商规范而不是凭感觉拼请求。 |
| **可视化**（`visualization-skill`） | **「结果怎么给人看」** | 统一 Demo 结果区体验：**左侧完整 JSON + 复制/下载**，**右侧可视化**（OCR 类在原图叠加框线、标签）；保证评测与演示时**看得懂、可存档、风格一致**。 |
| **删除 API / 下架**（`delete-api-platform-skill`） | **「干净下线」** | 按清单删除 `server.py` 路由、`demos/xxx.html`、`index.html` 菜单，并做**全仓检索验证**，防止只删菜单不删接口、或留下死路由；与**前端后端架构 Skill** 里「代码分布」一致，形成闭环。 |

### 6.2 协作顺序：平时怎么用这五个 Skill

- **新增一个厂商 Demo**：先 **架构 Skill** 定位置 → **厂商 API Skill** 定鉴权与调用模式 → 按 **集成 Skill** 五步改 `server.py` / `demos/` / `index.html` → 结果区按 **可视化 Skill** 实现左 JSON / 右图。  
- **只换接口、不换展示形态**：仍以 **集成 Skill** 为主线，**厂商 API Skill** 替换调用细节。  
- **不再展示某能力**：用 **删除 API Skill** 下架，必要时结合架构 Skill 确认是否只剩一个厂商 Demo、能否收缩鉴权与常量。

### 6.3 和业务价值的关系

五个 Skill 把「写脚本式的一次性验证」变成 **可重复、可审查、可交接的标准动作**：产品侧得到稳定体验（菜单、左 JSON 右图、复制下载），技术侧减少口头传话与漏改，组织侧便于推广同一套 **「接入—展示—下架」** 方法。

## 7. 技术架构（业务视角）

| 模块 | 作用 |
|------|------|
| **后端** | Python Flask 应用（`server.py`），提供页面与静态资源路由，以及各 Demo 对应的 API 转发与业务逻辑。 |
| **前端** | 入口页为侧栏 + iframe；各 Demo 为 `demos/` 下独立 HTML 页面，样式与交互遵循项目内可视化约定。 |
| **知识与规范** | `skills/` 下上述五个 Skill 及附带的 `*.md`，与代码一一对应，保证**接入方式一致、可交接**。 |

*注：不涉及强制业务中台或账号体系；本仓库侧重「集成与演示」，生产环境部署与安全策略由使用方自行评估。*

---

## 8. 典型使用场景

- **效果验证**：同一批样本在不同接口或不同参数下的快速对比。  
- **调研与选型**：在统一界面中体验厂商能力，减少临时脚本差异带来的偏差。  
- **汇报与演示**：左 JSON 右图，便于说明「输入—输出—可视化」链路。  
- **协作 onboarding**：新同学通过菜单与文档即可理解平台边界，再按需阅读 Skill 做二次开发。
